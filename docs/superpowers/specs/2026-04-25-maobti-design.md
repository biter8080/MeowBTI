# 猫BTI 核心测试版设计文档

## 1. 目标

本设计文档用于定义 `猫BTI` 离线 H5 Canvas 互动作品的首版实现边界。首版只聚焦核心链路：

- 首页进入
- `24` 题答题流程
- `16` 型主猫格结果页
- `E/P` 辅助副文案
- 离线分享图生成与保存

首版目标不是一次性做成完整测试平台，而是先做出一个可直接离线运行、可以完整体验、具备分享传播能力的核心测试版本。

## 2. 范围

### 2.1 本次包含

- `index.html` 作为唯一入口
- 移动端竖屏优先适配
- 首页、测试页、结果页、分享图预览层
- 基于文档整理得到的 `24` 道题与 `16` 型结果配置
- `E` 情绪表达度、`P` 压力防御方式的副文案拼装
- 本地 `localStorage` 进度恢复与最近结果缓存
- 轻量 Canvas 背景氛围
- 本地 Canvas 分享图导出
- 初始化和运行时错误兜底文案：`哎呀，出错了，请重启试试吧~`

### 2.2 本次不包含

- 图鉴真实收集系统
- 猫薄荷社区内容流
- 小游戏
- 多套测试切换
- 外部分享 SDK
- 网络请求、外链跳转、第三方资源加载

### 2.3 后续扩展预留

- 新增测试系列时，复用现有题库配置结构
- 结果对象补接真实 meme 图素材
- 插入图鉴页或更多结果页操作区
- 增加历史结果浏览

## 3. 技术约束

- 必须纯离线运行
- 禁止 `fetch`、`XMLHttpRequest`、`axios`、`WebSocket` 等任何网络请求
- 禁止外部 CDN、外部字体、外部图片与音频
- 所有资源都保存在项目目录内，并通过相对路径引用
- 入口必须为 `index.html`
- 界面不能出现横向滚动条
- 总体资源需要保持轻量，适合最终控制在 `8MB` 以内
- 必须支持在主流手机竖屏上完整显示

## 4. 总体方案

首版采用 `单页状态机 + DOM 内容层 + Canvas 背景/分享图` 的混合方案。

### 4.1 方案选择

采用单页应用的原因：

- 首版功能链路清晰，单页切换更轻量
- 离线环境更稳，避免多页面资源切换成本
- DOM 更适合长文本题目和结果文案阅读
- Canvas 更适合做动态氛围与分享图绘制

### 4.2 页面状态

应用包含以下状态：

1. `home`
2. `quiz`
3. `result`
4. `share-overlay`

其中 `share-overlay` 是结果页上的弹层状态，不作为独立路由页面存在。

## 5. 信息架构与流程

### 5.1 首页

首页展示以下内容：

- `icon.png` 品牌标识
- 标题与副标题
- `开始测试` 主按钮
- 轻量说明文案
- 非核心说明入口，如 `结果仅供娱乐`

用户点击 `开始测试` 后进入测试页。

### 5.2 测试页

测试页采用单题单卡结构：

- 顶部题号与进度条
- 中部题干卡片
- 下方四个选项按钮
- 返回上一题按钮

交互规则：

- 点击选项后自动进入下一题
- 用户可以返回上一题修改答案
- 答完最后一题后立即进入结果页

### 5.3 结果页

结果页统一包含：

- 结果标题
- 一句话标签
- 文字卡片型主视觉
- 人格介绍
- `E/P` 辅助副文案
- `生成分享图`
- `重新测试`

首版不接真实 meme 图，采用文字卡片、配色、装饰元素与品牌图形成可截图感。

### 5.4 分享预览层

结果页触发分享时弹出预览层，包含：

- 生成后的分享图画布预览
- `保存图片`
- `关闭`

保存逻辑使用浏览器本地下载能力，不触发任何外部跳转。

## 6. 数据设计

### 6.1 题库配置

题库使用配置对象统一管理，建议放入 `js/data.js`。

每道题至少包含：

- `id`
- `dimension`
- `prompt`
- `options`
- `scoreMap`
- `themeTag`

说明：

- `dimension` 对应六个维度之一
- `options` 为四个选项文本
- `scoreMap` 固定映射 `A/B/C/D -> +2/+1/-1/-2`
- `themeTag` 用于后续做视觉标记或分组扩展

### 6.2 结果配置

`16` 型结果统一配置，每项至少包含：

- `id`
- `typeCode`
- `name`
- `tagline`
- `description`
- `shareText`
- `catnipText`
- `collectionText`
- `memeKey`
- `accentColor`
- `cardStyle`

首版虽然不展示真实 meme 图，但保留 `memeKey` 作为后续素材接入字段。

### 6.3 辅助副文案配置

辅助副文案使用四种组合：

- `E+P+`
- `E+P-`
- `E-P+`
- `E-P-`

结果页根据 `E` 和 `P` 的正负组合动态读取一条副文案。

### 6.4 本地存储结构

建议使用以下键：

- `maobti.currentAnswers`
- `maobti.currentScores`
- `maobti.currentQuestionIndex`
- `maobti.lastResult`
- `maobti.resultHistory`

其中 `resultHistory` 可选，仅保留最近 `5` 条以内的简表记录。

## 7. 判定规则

### 7.1 六维累计

每答一题，将该题分值累计到对应维度：

- `S` 社交靠近度
- `E` 情绪表达度
- `A` 行动节奏感
- `O` 秩序与掌控感
- `D` 自我展示欲
- `P` 压力防御方式

### 7.2 主结果判定

以下四个主轴决定 `16` 型主猫格：

- `S`
- `A`
- `O`
- `D`

判定规则：

- 分数 `>= 0` 判定为正向
- 分数 `< 0` 判定为反向

最终拼成类型编码，例如：

- `S+A+O+D-`
- `S-A-O-D+`

再用该编码映射到具体猫格配置项。

### 7.3 平局处理

当主轴分数等于 `0` 时，统一判定为正向。

原因：

- 首版使用固定题量和对称分值，平局可能出现
- 统一规则可以避免边界结果不稳定
- 偏正向的结果在传播层面更容易形成轻松有趣的体验

### 7.4 辅助文案判定

`E` 与 `P` 轴同样按正负划分：

- 分数 `>= 0` 归正向
- 分数 `< 0` 归反向

组合后追加一条补充副文案，不改变主猫格结果。

## 8. UI 与视觉策略

### 8.1 布局方式

采用三层结构：

- 背景层：Canvas
- 内容层：DOM
- 弹层：分享图预览与错误提示

主容器宽度规则：

- `width: min(100vw, 480px)`
- 居中显示
- 全程禁止横向滚动

### 8.2 视觉方向

首版采用 `文字卡片型主视觉`：

- 复用现有 `icon.png`
- 主色使用暖黄、奶咖、粉色点缀
- 每个结果通过标题、标签、短句、强调色形成差异
- 不强行制作伪真实 meme 图

### 8.3 Canvas 职责

Canvas 只承担两部分职责：

1. 页面背景氛围
2. 分享图生成

Canvas 不承载主阅读内容，避免移动端文本适配复杂度过高。

### 8.4 交互节奏

- 答题切换保持干脆
- 选项反馈明显
- 减少复杂动画，优先保证轻快与稳定

## 9. 文件结构

首版建议结构如下：

```text
work_hks/
  index.html
  icon.png
  js/
    data.js
    app.js
    canvas.js
  css/
    style.css
  docs/
    superpowers/
      specs/
        2026-04-25-maobti-design.md
```

说明：

- 当前目录中的策划资料后续可由用户自行清理
- 实际可运行产物不依赖这些策划文档

## 10. 模块职责

### 10.1 `index.html`

- 提供应用根节点
- 引入样式、脚本与基础结构
- 放置错误兜底容器

### 10.2 `js/data.js`

- 输出题库配置
- 输出结果配置
- 输出辅助副文案配置

### 10.3 `js/app.js`

- 初始化应用
- 管理页面状态切换
- 渲染首页、答题页、结果页
- 处理答题、回退、评分、结果计算
- 管理本地存储恢复
- 处理错误兜底

### 10.4 `js/canvas.js`

- 渲染轻量背景动画
- 生成分享图
- 处理 `icon.png` 加载与降级策略

### 10.5 `css/style.css`

- 实现移动端布局
- 定义品牌色与卡片样式
- 提供按钮、题卡、结果卡、弹层的视觉规范

## 11. 错误处理

应用初始化和关键逻辑必须用 `try-catch` 包裹。

错误策略如下：

- 初始化失败时显示 `哎呀，出错了，请重启试试吧~`
- 图片资源加载失败时降级为纯文字模式
- `localStorage` 读写失败时不阻断主流程
- 分享图生成失败时给出友好提示并允许用户返回结果页

## 12. 测试与验收

### 12.1 功能验证

必须验证：

- 首页能正常进入测试
- `24` 题可完整答完
- 返回上一题后分值与答案能正确更新
- `16` 型主猫格映射正确
- `E/P` 副文案组合正确
- 分享图可生成并保存
- 重新测试可重置当前流程

### 12.2 适配验证

必须验证：

- 手机竖屏下无横向滚动
- `360px` 到 `430px` 宽度区间可正常显示
- 长题干和长结果文案不会溢出卡片

### 12.3 运行验证

必须验证：

- 直接打开 `index.html` 可离线运行
- 不依赖开发服务器
- 页面不发起任何网络请求

## 13. 实现顺序

推荐开发顺序：

1. 整理题库和结果配置到 `js/data.js`
2. 搭建 `index.html` 与 `css/style.css`
3. 完成 `js/app.js` 的状态机与判题逻辑
4. 完成 `js/canvas.js` 的背景与分享图
5. 验证移动端适配、错误处理与本地存储恢复

## 14. 风险与边界

### 14.1 主要风险

- 题目与结果配置量较大，手工整理时容易出现映射错误
- 分享图在不同浏览器上的下载体验存在差异
- 若 Canvas 背景动画过重，低端手机上可能掉帧

### 14.2 控制策略

- 配置数据尽量结构化，避免散落在逻辑中
- 分享图以静态一次性绘制为主，不做复杂合成
- 背景动画保持低粒度和低频更新

## 15. 结论

首版 `猫BTI` 将以“核心测试版”方式落地，使用单页状态机组织首页、答题页、结果页和分享层，用 DOM 承担阅读与交互，用 Canvas 承担轻氛围和分享图导出。该设计满足纯离线、轻量化、可移动端运行、可继续扩展的目标，并为后续增加图鉴、更多测试系列和结果素材保留接口。
