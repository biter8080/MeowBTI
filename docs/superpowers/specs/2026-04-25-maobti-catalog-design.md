# 猫BTI 图鉴解锁功能设计

## 1. 目标

本设计用于在现有 `猫BTI` 离线单页应用中加入 `图鉴解锁` 能力，让用户在完成测试后自动收录当前结果，并在首页进入 `我的图鉴` 查看已解锁与未解锁猫格。

本次设计只解决以下问题：

- 测试结果自动保存为图鉴解锁状态
- 首页新增 `我的图鉴` 入口
- 提供独立图鉴页展示 `16` 种猫格收集进度
- 未解锁猫格使用问号封面占位
- 点击图鉴卡可查看详情，未解锁卡给出提示

本次不引入账号体系、云端同步或历史时间线，只做本地离线收藏册。

## 2. 范围

### 2.1 本次包含

- 首页新增 `我的图鉴` 按钮
- 首页展示 `已解锁 X / 16` 收集进度
- 测试结束后自动解锁当前结果
- 本地图鉴数据持久化到 `localStorage`
- 独立图鉴页
- 图鉴详情弹层
- 未解锁卡问号封面
- 未解锁点击提示
- 手机端优先的图鉴网格适配

### 2.2 本次不包含

- 每次测试历史记录
- 图鉴分享功能
- 图鉴筛选、排序、搜索
- 社区联动
- 跨设备同步
- 服务器存储

## 3. 现状与约束

当前项目已具备以下基础：

- 单页状态机，主要状态为 `home`、`quiz`、`result`
- 使用 `localStorage` 保存答题进度与最近结果
- 结果页已接入本地 meme 图
- 结果页与分享层已具备弹层模式，可复用为图鉴详情层

新增图鉴功能时必须继续遵守以下约束：

- 保持纯离线运行
- 不引入网络请求
- 不新增复杂路由系统
- 手机端不能出现横向滚动
- 不影响已有答题、结果、分享流程

## 4. 方案选择

本次采用 `方案 A：独立图鉴页 + 自动解锁 + 详情弹层`。

### 4.1 选择原因

- 与用户要求的“我的图鉴”“解锁图鉴”“点击图片查看图鉴”最匹配
- `16` 张卡更适合放在独立页面中滚动浏览，不适合全部塞入弹层
- 与现有单页状态机兼容，只需扩展一个新视图状态
- 自动解锁比“手动保存到图鉴”更顺滑，不增加额外操作负担

### 4.2 明确不选的方案

- 不做“仅弹层图鉴”，因为移动端容纳 `16` 张卡可用性较差
- 不做“历史记录优先”，因为本轮需求重点是图鉴解锁，不是测试时间线

## 5. 信息架构与交互

### 5.1 首页

首页在现有结构基础上新增：

- `我的图鉴` 按钮
- `已解锁 X / 16` 进度文案

交互规则：

- 点击 `开始测试` 进入答题页
- 点击 `我的图鉴` 进入图鉴页

### 5.2 结果页

结果页保持现有主体结构，但增加一条轻提示：

- `已收录进你的图鉴`

保存规则：

- 当用户完成最后一题并生成结果后，立即执行自动解锁
- 若该结果已解锁，不重复写入
- 自动解锁不需要用户确认，不新增额外按钮

### 5.3 图鉴页

图鉴页是新的独立视图，包含：

- 顶部标题 `我的图鉴`
- 返回首页按钮
- 当前收集进度
- `16` 张图鉴卡组成的网格

每张图鉴卡对应一个结果项：

- 已解锁：显示真实结果图和名称
- 未解锁：显示问号封面和 `???`

### 5.4 图鉴详情层

点击已解锁卡后，打开详情弹层，内容包括：

- 结果图片
- 结果名称
- tagline
- description

弹层只承担查看职责，不在本轮加入分享、跳转或额外收集动作。

### 5.5 未解锁提示

点击未解锁卡后，不打开完整详情，而是显示轻量提示：

- `这只猫还没解锁`

提示形式优先复用现有弹层体系，保持视觉一致，不新增浏览器原生 `alert()`。

## 6. 数据设计

### 6.1 存储键

新增图鉴专用存储键，例如：

- `maobti.collection`

### 6.2 存储结构

图鉴只保存解锁状态，不保存重复次数和历史记录，建议结构如下：

```json
{
  "unlockedResultIds": ["01", "05", "12"]
}
```

规则说明：

- `unlockedResultIds` 使用结果对象稳定 `id`
- 同一个 `id` 只能出现一次
- 读取失败时回退为空图鉴
- 写入失败时不阻断结果页渲染

### 6.3 解锁逻辑

自动解锁流程：

1. 结果页完成类型计算
2. 取得当前结果 `id`
3. 读取图鉴数据
4. 若该 `id` 尚未存在，则写入
5. 返回最新图鉴状态用于界面渲染

该逻辑必须幂等，同一结果重复出现不能造成重复数据。

## 7. 视图与模块设计

### 7.1 状态层

在现有页面状态基础上新增：

- `collection`

用于表示图鉴页正在显示。

### 7.2 `state.js`

新增图鉴相关职责：

- 创建空图鉴状态
- 读取图鉴数据
- 保存图鉴数据
- 解锁某个结果
- 计算已解锁数量

这些函数应与答题进度存储分开，避免耦合到 `session` 结构。

### 7.3 `main.js`

新增以下主流程职责：

- 初始化加载图鉴数据
- 结果页渲染前或渲染时执行自动解锁
- 处理首页进入图鉴页
- 处理图鉴页返回首页
- 处理图鉴卡点击
- 处理图鉴详情层与未解锁提示层的打开关闭

图鉴逻辑应尽量围绕现有事件代理模式扩展，不新增分散绑定。

### 7.4 `templates.js`

新增或扩展以下模板：

- 首页图鉴入口与进度文案
- 图鉴页模板
- 图鉴详情弹层模板
- 未解锁提示弹层模板

模板层只负责结构输出，不负责读写本地存储。

## 8. 视觉与适配策略

### 8.1 图鉴页布局

图鉴页采用手机优先网格布局：

- 手机默认 `2` 列
- 卡片宽度流式缩放
- 页面整体只允许纵向滚动

在更宽设备上可增强列宽或间距，但不切换为复杂桌面布局。

### 8.2 已解锁卡

已解锁卡包含：

- 结果图缩略图
- 结果名称
- 已解锁视觉状态

需要与结果页现有图片风格保持一致，避免像另一个产品。

### 8.3 未解锁卡

未解锁卡不依赖新增图片资源，而是使用样式化占位卡：

- 中央大号 `?`
- 下方 `???`
- 更轻的底色、边框或透明度

这样可以避免额外素材管理，也更稳定。

### 8.4 详情层

详情层延续现有 overlay 风格：

- 居中卡片
- 遮罩点击关闭
- 手机端不超出可视宽度

详情层优先复用已有按钮区与卡片样式规则，减少视觉分叉。

## 9. 错误处理

- 图鉴存储读取失败时回退为空图鉴，不影响首页和结果页
- 图鉴存储写入失败时保持结果页可正常显示
- 已解锁卡图片加载失败时，可降级为无图文字卡，但仍保留已解锁状态
- 未解锁卡点击不能抛错，至少显示友好提示
- 若图鉴数据出现非法 `id`，渲染时忽略非法项

## 10. 测试与验收

### 10.1 状态测试

必须验证：

- 无图鉴数据时能安全回退为空状态
- 解锁结果后会写入正确的 `id`
- 重复解锁同一结果不会重复写入
- 图鉴计数正确

### 10.2 模板测试

必须验证：

- 首页出现 `我的图鉴`
- 首页能显示已解锁数量
- 图鉴页能输出 `16` 张卡
- 未解锁卡显示问号占位
- 已解锁卡显示真实结果信息
- 详情层输出结果名称与说明

### 10.3 流程测试

必须验证：

- 测试完成进入结果页时自动解锁当前结果
- 返回首页后进图鉴能看到刚解锁的卡
- 点击已解锁卡可查看详情
- 点击未解锁卡显示未解锁提示

### 10.4 适配验收

必须保证：

- 手机端图鉴页无横向滚动
- 图鉴卡不会因标题或图片溢出容器
- 详情层在主流手机宽度下可完整阅读和关闭

## 11. 实现顺序

推荐实现顺序：

1. 扩展图鉴本地存储与状态函数
2. 扩展首页入口与图鉴页模板
3. 接入结果页自动解锁
4. 接入图鉴详情层与未解锁提示
5. 补齐样式、测试与 bundle 构建

## 12. 结论

本次图鉴功能采用“首页入口 + 自动解锁 + 独立图鉴页 + 详情弹层”的方案，在不改变离线单页架构的前提下，为现有 `猫BTI` 增加一个轻量、明确、可持续扩展的收藏层。该方案满足用户提出的保存测试结果、逐步解锁图鉴、未解锁显示问号和点击图片查看详情的核心需求，同时保持移动端可用性和实现复杂度可控。
