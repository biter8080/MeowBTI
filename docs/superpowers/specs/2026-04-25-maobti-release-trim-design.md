# 猫BTI 极简发布版清理设计

## 1. 目标

把当前 `猫BTI` 工程收缩为仅保留可直接运行的静态成品目录，不考虑后续维护、开发、测试或再次构建。

本次目标只有两点：

- 页面仍可直接打开并运行
- 在不压缩图片资源的前提下，尽可能删除所有非运行期文件

## 2. 范围

### 2.1 保留内容

最终目录只保留以下运行必需文件：

- `index.html`
- `css/style.css`
- `js/app.bundle.js`
- `resources/personalities-main/*`
- `icon.png`

### 2.2 删除内容

以下内容全部删除：

- `.superpowers/`
- `docs/`
- `tests/`
- `scripts/`
- `package.json`
- `package-lock.json`
- `index.dev.html`
- `js/main.js`
- `js/core/`
- `js/data/`
- `js/ui/`

## 3. 实现方案

### 3.1 入口切换

当前 `index.html` 使用：

- `./js/main.js`

清理前先改为：

- `./js/app.bundle.js`

这样运行时只依赖单个 bundle，不再依赖源码模块树。

### 3.2 目录收缩

在入口切换完成并验证通过后，删除所有开发期文件，只留下成品目录结构。

本次不做以下事情：

- 不压缩 `PNG`
- 不改图片尺寸
- 不新增发布脚本
- 不保留任何测试、文档或源码备份

## 4. 风险与取舍

- 删除后项目将不再适合继续开发
- 无法再使用现有测试和构建流程
- 若后续需要改动，只能基于 bundle 成品重新反向整理，成本较高

这些取舍与本轮“只留运行文件、怎么小怎么来”的目标一致。

## 5. 验收标准

完成后必须满足：

- 打开 `index.html` 可正常进入首页
- 完成答题后可进入结果页
- 结果页 `我的图鉴` 可打开图鉴页
- 分享图功能仍可正常生成
- 页面无缺失脚本、缺失样式或缺失图片导致的运行错误

## 6. 结论

本次采用“切到 bundle 后原地瘦身”的方案，在不压缩图片的前提下，把工程压缩为最小可运行静态发布包。这是当前约束下体积最小且风险最低的清理方式。
